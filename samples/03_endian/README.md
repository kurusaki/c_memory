## 動画シナリオ

今回は、変数はメモリ上でどう保存されているのかを見ていきます。

C言語には、 `char`、`short`、`int`、`long`  など、さまざまな型があります。

同じ `20` という値でも、型が違うと、メモリ上で使われるサイズが変わります。

今回は、前回作成した **mdump 関数** を使って、実際にメモリの中を見てみましょう。


### 今回使うファイル

今回のサンプルでは、3つのファイルを使います。

- mdump.c   : メモリダンプ関数の実装
- mdump.h   : mdump関数の宣言
- sample1.c : 今回のサンプルプログラム

#### mdump.c

```c
#include <stdio.h>
#include <stddef.h>
#include <ctype.h>

void mdump(const void *addr, size_t size)
{
    unsigned char *p = (unsigned char *)addr;

    for (size_t i = 0; i < size; i += 16) {

        printf("%p : ", p + i);

        /* HEX表示 */
        for (size_t j = 0; j < 16; j++) {

            if (i + j < size) {
                printf("%02X ", p[i + j]);
            } else {
                printf("   ");
            }
        }

        printf(" ");

        /* ASCII表示 */
        for (size_t j = 0; j < 16 && i + j < size; j++) {

            unsigned char c = p[i + j];

            if (isprint(c)) {
                printf("%c", c);
            } else {
                printf(".");
            }
        }

        printf("\n");
    }
}
```

このファイルには、メモリの内容を表示する `mdump` 関数が実装されています。

今回は、この関数を使って変数の中身を見ていきます。


#### mdump.h

```c
#ifndef MDUMP_H
#define MDUMP_H

#include <stddef.h>

void mdump(const void *addr, size_t size);

#endif
```

ヘッダファイルには、他のソースファイルから使う関数の宣言を書きます。

`mdump.h` をインクルードすることで、`mdump` 関数を使えるようになります。

#### sample1.c

```c
#include <stdio.h>
#include "mdump.h"

int main(void) 
{
    char  c = 20;
    short s = 20;
    int   i = 20;
    long  l = 20;

    printf("===== char  (20) : 1バイト =====\n");
    mdump(&c, sizeof(c));

    printf("===== short (20) : 2バイト =====\n");
    mdump(&s, sizeof(s));

    printf("===== int   (20) : 4バイト =====\n");
    mdump(&i, sizeof(i));

    printf("===== long  (20) : 8バイト =====\n");
    mdump(&l, sizeof(l));

    return 0;
}
```

今回使うプログラムはこちらです。

まず、`char`、`short`、`int`、`long` それぞれの変数に、同じ `20` を代入しています。

そして、`mdump` を使って、その変数が確保しているメモリ領域を表示しています。

ここで使っている `sizeof` は、その変数が何バイト使っているかを取得する演算子です。

つまり、変数の先頭アドレスから、その変数のサイズ分だけメモリを表示しています。



### コンパイルして実行する

コンパイルして実行してみましょう。

今回は、`sample1.c` だけではなく、`mdump.c` も一緒にコンパイルします。

```bash
clang sample1.c mdump.c -o sample1
```

これは、
`sample1.c` と `mdump.c` をまとめてコンパイルして、
`sample1` という実行ファイルを作る、という意味です。


実行します。
```bash
./sample1
```

【実行結果】
```text
===== char  (20) : 1バイト =====
0x16fa96d8b : 14                                               .
===== short (20) : 2バイト =====
0x16fa96d88 : 14 00                                            ..
===== int   (20) : 4バイト =====
0x16fa96d84 : 14 00 00 00                                      ....
===== long  (20) : 8バイト =====
0x16fa96d78 : 14 00 00 00 00 00 00 00                          ........
```

実行すると、このような結果になります。

一見すると不思議ですね。

`20` を代入したはずなのに、どれも `14` と表示されています。

さらに、型によって表示されるバイト数も違います。

では、一つずつ見ていきましょう。


### char

まずは `char` です。

```c
char  c = 20;
```

`char` は1バイトの型です。

そのため、メモリダンプでも1バイトだけ表示されています。

```text
===== char  (20) : 1バイト =====
0x16fa96d8b : 14                                               .
```

### short

続いて `short` です。

```c
short s = 20;
```

`short` は2バイトです。

そのため、2バイトが表示されています。

```text
===== short (20) : 2バイト =====
0x16fa96d88 : 14 00                                            ..
```


### int

次は `int` です。

```c
int   i = 20;
```

この環境では `int` は4バイトです。

そのため、4バイトが表示されています。

```text
===== int   (20) : 4バイト =====
0x16fa96d84 : 14 00 00 00                                      ....
```


### long

最後は `long` です。

```c
long  l = 20;
```

Macでは `long` は8バイトです。

そのため、8バイトが表示されています。

```text
===== long  (20) : 8バイト =====
0x16fa96d78 : 14 00 00 00 00 00 00 00                          ........
```

このように、同じ `20` を保存していても、型によって使うメモリの大きさが違うことが分かります。


### 20なのに、なぜ14？

ここで、一つ疑問があります。

`20` を代入したのに、なぜ `14` なのでしょうか？

これは、`mdump` がメモリの中身を 16進数 で表示しているからです。

私たちが普段使っている数字は10進数です。

一方、メモリダンプでは16進数で表示するのが一般的です。

そのため、

10進数では `20`

16進数では `14`

と表現します。

メモリの世界では、このように16進数がよく使われます。

### もう一つ気になること

なぜ、`00 00 00 14` ではないのでしょうか？

実は、この並び方にも理由があります。

この理由は、次回解説する **エンディアン** という仕組みに関係しています。
